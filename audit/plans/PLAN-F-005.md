## Approach

- **zod, hand-rolled two-schema split.** zod is NOT a dep — add it **pinned exact** (current latest stable, verify on npm; follows v5 §Typing which F-014 applies repo-wide; note it's the first pinned dep). Update **root** `pnpm-lock.yaml` (CI `--frozen-lockfile`). Do NOT add to knip `ignoreDependencies` (it's imported).
- **Server/client separation** because `NEXT_PUBLIC_*` are build-inlined and client-reachable: server schema carries secrets + `import 'server-only'` (blocks accidental client import; Next provides the pkg — if tooling flags it unlisted, add `server-only` pinned). Client schema all-optional so the browser parse never throws.
- **Testable singleton:** export pure `parseServerEnv(source=process.env)` (safeParse→throw listing `issue.path` names) and `export const env = parseServerEnv()`. Negative tests call the pure fn with bad fixtures; the eager singleton import stays safe in vitest via `test.env`.
- **Base-URL reconciliation = one fallback, not eight:** `BASE_URL = clientEnv.NEXT_PUBLIC_BASE_URL ?? 'https://grabcaramel.com'`. Replacing `process.env.NEXT_PUBLIC_BASE_URL || 'https://grabcaramel.com'` is value-identical (zero behavior change) at marketing/auth pages; at `auth.ts`/`auth/client.ts` the unset-fallback shifts `localhost:3000`→prod URL (only fires when the var is entirely unset — never in dev; prod URL is the correct default).
- **Rejected:** `@t3-oss/env-nextjs` (adds a heavier dep + a second convention; hand-rolled zod matches "one zod module"). Per-var lazy validation (keeps the 3 divergent strategies F-005 is killing). `NEXT_PUBLIC_BASE_URL` required (risks a client/prod-build crash; optional+single-fallback still reconciles).

## Rollback

One commit `fix(F-005): zod-validated env module + regenerated .env.example`. Internal checkpoints at steps 1,3,4,6,7 (git stash/scratch commits) so a failed later step restarts without losing pins. Safe revert: `git revert` the single commit — reintroduces `process.env` reads and the old `.env.example`; no data/schema migration, fully reversible. If prod boot fail-fasts on a genuinely-missing var post-deploy, that is correct behavior (set the var) — not a rollback trigger.
